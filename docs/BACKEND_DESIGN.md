# バックエンド設計書 (v0.2)

作成日: 2026-07-13 / 最終更新: 2026-07-14 (v0.2)

## 1. 位置づけ・スコープ

- リプレイ共有(短縮じゅもんの発行・解決)を担うサーバーサイドの方式設計。
- スコープ: **シミュレーターのリプレイ共有のみ**。保存はユーザーの明示的な操作(「リプレイをコピー」相当)のときだけ行い、**シミュレーターでは明示的な保存操作以外のプレイ結果収集は行わない**(2026-07-13 確定)。
- 計算機モードのプレイ結果収集は本書の将来拡張(§10)として枠だけ確保する(収集の主対象は計算機側。2026-07-13)。
- 関連文書: MOBILE_UI_DESIGN.md(D38/D39/§5.14 リプレイコード入力画面)、ARCHITECTURE.md(A4 乱数消費順=リプレイ互換の根幹)、config-compact-codec(圧縮コーデック本体)。

## 2. 全体構成

```
[ iOS / Android アプリ・Web版 ]
        │  共有時のみ POST /api/replays(リプレイデータ)
        ▼
[ Cloudflare Workers ]  … API・形式検証・レート制限(ゲームロジックは持たない)
        │
        ▼
[ Cloudflare D1 ] ──(Cron Trigger 日次)──▶ [ Cloudflare R2 ](バックアップ)
```

- **Workers はゲームロジックを一切持たない**。検証は形式(文字種・長さ・チェックサム・デコード可能性)のみ。これによりアプリの仕様変更・スキーマ改版でサーバー改修が不要になる。
- Web版の静的配信は現行の GitHub Pages を継続(Cloudflare Pages への移行は任意。§11-4)。

## 3. リプレイデータの表現形式(3層)

| 用途 | 形式 | 長さの目安 |
|---|---|---|
| 正準データ | config-compact-codec の rank(bigint)+スキーマ版数 | - |
| ユーザー向け完全じゅもん | ひらがな71進: バージョン1文字+**固定長ペイロード**+チェックデジット1文字 | 約45文字(固定) |
| サーバー送受・格納 | 同一 rank の **Base64url 表現**+版数フィールド(JSON) | 約45文字(64進) |

- **完全じゅもん(ひらがな)はフォールバック用の永続フォーマット**: サーバー通信のタイムアウト・障害時にアプリが出力し、サーバーが消滅しても復号できる(オフライン完結)。
- **サーバー送受・格納は Base64url**(2026-07-13 ユーザー提案): サーバー経路のフォーマットはユーザーの視認性・SNS投稿の利便性を考慮する必要がないため、一般的な手法を使う。codec はアルファベット差し替え可能なので、**同一の rank から Base64url 版を同一実装で生成**でき、変換層は発生しない。
- **完全じゅもんは可変長のまま**とする(2026-07-14 確定。固定長化・codec の payloadLength オプションは不要): 完全/短縮の判別は §4 の先頭文字規則で行う。コード長は**実際のターン数に比例**し、実測で 1ターン=11文字・10ターン=22文字・30ターン=45文字・58ターン=78文字(版数・チェックデジット込み)。

### 3.1 スキーマ(D38 確定分)

| 項目 | 範囲 | 備考 |
|---|---|---|
| レシピID | 整数 0〜999 | recipes.csv に id 列を追加。正式版レシピ更新時にソート後の行ベースで採番(タイミングはユーザー連絡。互換性維持は不要) |
| 針の種類 | 7種(SPEC §3.4) | |
| 針の★ | 0〜3 | |
| 職人Lv | 整数 1〜80 | Lv設定なし時は80 |
| コツ | あり/なし | |
| シード | 整数 0〜2^32−1 | |
| 行動列 | 可変長配列・**maxItems=64**・**処理順の先頭**(schema の order 指定) | 各要素=特技(§3.3 の20種)+アンカー位置(0〜8)。対象を取らない特技はアンカー省略(dependsOn)。理論上限58ターン(虹以外でぬう連打+消費0の必殺1回)+アップデートでの集中力上昇余裕(たかだか+40程度)で64に確定(2026-07-13) |

- **行動列は処理順の先頭に置く**(2026-07-14 訂正): 当初「最後尾」としていたのは誤り。ランキング方式では**配列より処理順で前にある項目が値を1つ進むたびに配列の理論上限(maxItems分)の組み合わせ数が乗算される**ため、行動列を最後尾に置くとシードが実質必ず非0であることから**全コードが maxItems=64 相当の85文字固定**になってしまう(実測)。先頭に置けば maxItems の余裕はコード長に影響せず、長さは実ターン数に比例する(上記実測値)。

含めないもの: 開幕特殊効果の発動有無(シードから再現)、表示系設定(C1/C3)、特技ボタン配置(C2)、開始集中力(導出値)。

## 4. 短縮じゅもん仕様

- **8文字 = ランダムキー7文字+チェックデジット1文字**(すべてひらがな71種。2026-07-13 確定)。
- **バージョン文字は持たない**: 短縮じゅもんは不透明な参照キーであり、スキーマ版数はサーバーの行(schema_version)が持つため不要。**全文字が実質一様ランダムになり、先頭が毎回同じ文字になる問題(隠蔽の要望)は構造的に発生しない**。
- キー空間 71^7 ≈ 9.0×10^12(先頭文字の除外(下記)で 70×71^6 ≈ 8.9×10^12)。CSPRNG で採番し、衝突時は再採番。
- **完全/短縮の判別は「文字数+先頭文字」**(2026-07-14 確定): 完全じゅもんは可変長のため、まれに8文字ちょうどになり得る(全項目が先頭値・1ターン・シード約20億などの極端例で8文字を実測構築)。このため文字数だけでは判別が確定しない。**完全じゅもんの先頭は常に版数文字(v1=「あ」)**である性質を使い、**短縮キーの1文字目は版数として割当済みの文字を除いた集合(v1時点で70種)から採番**する。判別規則: **8文字かつ先頭が版数文字でない=短縮(サーバー照会)/それ以外=完全(ローカル復号)**。可変長を保ったまま判別が一意に決まり、codec の変更も不要。
- チェックデジットにより、誤入力の大半は**サーバー照会前にクライアント側で検出**できる(通信を発生させない)。

## 5. API 仕様(案)

| API | 内容 |
|---|---|
| `POST /api/replays` | body `{ v: 1, data: "<base64url>" }` → `201 { key: "<短縮じゅもん8文字>" }`。検証: 文字種・サイズ上限(例: 256バイト)・チェックサム |
| `GET /api/replays/{key}` | → `200 { v: 1, data }` / `404` |

- レート制限: IP単位(POST 回数/日・GET 回数/分)。初期値は §11-5。Workers 内実装または Cloudflare WAF。
- UI との対応: チェックサム不一致(ローカル検出)・404 → 「じゅもんが ちがいます」系ダイアログ(§5.14)。ネットワークエラー・5xx → 通信エラー表示(再試行導線)。**保存時のタイムアウト・失敗 → 完全じゅもん(ひらがな45文字)を代わりに出力**(共有自体は常に成立する)。

## 6. データモデル(D1)

```sql
CREATE TABLE replays (
  key              TEXT PRIMARY KEY,   -- ランダム7文字(チェックデジットは含めない)
  data             TEXT NOT NULL,      -- base64url ペイロード
  schema_version   INTEGER NOT NULL,
  created_at       TEXT NOT NULL,
  last_accessed_at TEXT NOT NULL
);
CREATE TABLE config (
  name  TEXT PRIMARY KEY,              -- 'retention_days' など
  value TEXT NOT NULL
);
```

## 7. 保存期限

- **保持日数はサーバー側の設定として D1 の config テーブルに持つ**(2026-07-13 要望): デプロイ不要で運用中に変更できる。未設定=無期限。
- Cron Trigger(日次)が期限超過の行を削除。**期限の基準は created_at 固定**(2026-07-14 確定)。last_accessed_at 列は将来の分析・延命方式への変更余地として残す。
- 期限切れキーの照会は 404 → 「じゅもんの ほかんきげんが きれています」等の文言を §5.14 に追加予定。

## 8. 不正・悪用対策

- 形式検証+サイズ上限で任意データ置き場化を防止(データはゲーム設定値の列挙のみ)。
- レート制限で大量投稿・総当たりを抑止(キー空間 9×10^12 に対し列挙は実質不可能)。
- **PII を一切保存しない**(アカウントなし・ゲームデータのみ)。漏えい時の影響は実質なし。

## 9. バックアップ・復旧

- D1 標準の Time Travel(30日間の時点復元)。
- Cron Trigger で日次 R2 エクスポート(形式・保持世代数は §11-6)。
- 復旧手順: Time Travel で復元 → 直近エクスポートと突合。
