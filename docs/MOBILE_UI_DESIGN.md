## 3. 画面前提の検討パターン(未決定 — 論点表)

縦持ち固定を前提に、スマホアプリで一般的に考慮するパターン。各論点を決めてから画面設計に入る。

| # | 論点 | 一般的な考え方 | 決定 |
|---|---|---|---|
| S1 | 基準デザイン幅と下限 | 幅 360〜390dp を基準に設計し、下限を決める。事実上の下限は **360×640dp**(Android 普及帯)/ iOS は 375×667(iPhone SE)。下限を切り上げるほど設計は楽になるが対象端末が減る | 未 |
| S2 | アスペクト比の変動 | 同じ幅でも高さは 640〜900dp+ と大きく変わる(16:9〜21:9)。**どの領域を伸縮させるか**(例: ログ/余白を伸ばし、グリッド・ボタンは固定)を決めるのが縦持ちUIの要点 | 未 |
| S3 | セーフエリア | ノッチ・Dynamic Island・ホームインジケータ・ステータスバーを避ける **safe area inset** 対応。背景画像はセーフエリア外まで敷き、操作要素は内側に置くのが定石 | 未 |
| S4 | タッチターゲット寸法 | 最小 **44pt(iOS)/48dp(Android)**。マス・特技ボタンはこれ以上を確保。誤タップ防止の間隔も含めて設計 | 未 |
| S5 | 素材解像度 | 生成画像は表示サイズの **2〜3倍**で用意し縮小表示(高DPI対応)。ファイルサイズと相談 | 未 |
| S6 | OSフォントスケール | アクセシビリティの文字拡大にUIを追従させるか、ゲームUIとして固定するか。リッチUI(画像枠に文字を載せる)は**固定が無難** | 未 |
| S7 | タブレット | 縦持ち固定のまま「中央寄せ+余白(レターボックス)」か「拡大」か。対象外とするかも含めて決める | 未 |
| S8 | ダークモード | 対応するか、テーマを1つに固定するか。背景画像主体のリッチUIなら**固定が一般的** | 未 |
| S9 | キーボード表示時 | シード入力等でソフトキーボードが出たときのレイアウト崩れ(入力欄が隠れない)対策。検証系入力は別画面/モーダルに寄せると楽 | 未 |
| S10 | 横持ちの扱い | 縦固定なら横向きは単純ロック。iPad で縦固定にする場合はフルスクリーン指定が必要(ストア要件) | 未 |

## 4. 素材方針(生成画像)

### 4.1 パーツ分割の設計観点(差し替え考慮)

素材は「画面一枚絵」ではなく**部品単位**で管理し、個別差し替え・状態差分・多解像度化に耐える構成とする。

- **背景**: 画面全体の背景(セーフエリア外まで敷く)。画面種別ごと(プレイ画面/結果画面など)
- **パネル枠**: 状態表示・ログ等の枠。**9-slice(周囲を固定し中央を伸縮)**で可変サイズに対応できる形が望ましい
- **ボタン**: 特技ボタン・汎用ボタン。通常/押下/無効/選択中の**状態差分**をセットで用意。文字は画像に焼き込まず**テキストを重ねる**(特技追加・文言変更・多言語化に耐える)
- **マス**: 布マスの下地(状態色: 白/黄/緑/赤 は下地差分 or 色調変化のどちらで表すか)。しつけ枠・発光・対象ハイライトは**重ねる装飾パーツ**として分離
- **アイコン・バッジ**: ぬいパワー、布特性、必殺チャージ、無我など
- **エフェクト**: 吹き出し(ダメージ/会心/回復)、必殺演出など(静止画+CSS/コードアニメーションか、連番/スプライトか)

命名・格納規約(assets/ 配下のディレクトリ構成、状態差分のサフィックス等)は画面設計の確定後に定める。

### 4.2 権利面の線引き

- DQ10 の**画像・ロゴ・フォント・スクリーンショットの流用は不可**(著作権)。生成画像でも「DQ10 の UI を忠実に再現したもの」は依拠性が認められ得るためリスクがある。
- 「木枠+布地風」「和風RPGメニュー調」のような**雰囲気のレベルで寄せる**のは一般に問題ない。固有のデザイン(DQ特有のウィンドウ枠・書体・配色の組合せの再現)は避ける。

### 4.3 DQ10寄せの見え方確認(P3)

判断材料として、モック段階で「寄せない案」と「雰囲気を寄せた案」の2案を作成して比較する(生成画像 or HTMLモック)。比較後にトーンを決定し、以降の素材生成プロンプトの基準にする。

## 5. 表示項目インベントリ(ゼロベースの要不要確認)
