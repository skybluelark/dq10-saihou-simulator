## 1. 位置づけ

本書は実装に入る前の進め方(段階・順序・各段階で決めること)を定める。ゲーム仕様そのものは SPEC.md を正とする。

## 2. 将来構想(計画に織り込む前提)

初期リリースのスコープ外だが、以下を将来実施する意向があるため、初期の設計判断で道を塞がないこと。

| # | 将来構想 | 初期設計への影響 |
|---|---|---|
| W1 | スマホアプリ(Android/iPhone)への移植 | コアロジックをUIフレームワーク非依存の純TSパッケージとして分離。UIはタッチ操作・縦画面を前提に設計。移植戦略(Capacitor/PWA/React Native)を方式設計で選定。**UIは機能・リリース優先だったブラウザ版と異なり、ボタン画像・背景等を用いたリッチな見た目を志向する(2026-07-08 決定)** |
| W2 | Lv制限撤廃等の制限解除(職人レベル可変、コツ・パッシブ有無の切替) | SPEC §1.1 の実装前提をハードコードせず、**デフォルト値つきの設定オブジェクト**としてコアAPIに持たせる(初期UIでは露出しない)。レベル別集中力テーブル・特技習得Lvはデータ保持済み。実装時は**検証モード限定機能**とし、コツはレシピ別取得のためレシピ選択後にチェックボックス等で入力する(SPEC §1.1、2026-07-07 決定) |
| W3 | 実機プレイと並行使用する「計算機」機能(実機の途中状態を入力し、確率計算・行動提案を得る) | ゲーム状態(GameState)を**レシピ開始時からの遷移だけでなく、任意の途中状態から構築可能**なAPIにする。期待値・確率計算関数はF9ソルバーと共通基盤。状態はシリアライズ可能にする |

## 4. 各段階の詳細

### 4.1 非機能要件の確認

決めること(いずれも軽量な確認で足りる):

1. 対応ブラウザ・動作環境(PC優先か、当初からスマホブラウザも動作対象か — W1に関連)
2. オフライン動作の要否
3. 統計モードの試行回数規模(例: 10万回をどの程度の時間で回すか — コアの性能要件)
4. 設定・行動ログのローカル保存要否(localStorage等)

### 4.2 方式設計

論点一覧。各論点について推奨案を提示→承認の形で確定する。

| # | 論点 | 補足(将来構想の影響) |
|---|---|---|
| A1 | 技術スタックの確定(TypeScript+React+Vite案、テストはVitest) | W1を踏まえ、UI層の技術はモバイル移植戦略(A8)とセットで判断する |
| A2 | モジュール構成: `core/`(純関数ゲームエンジン)・`data/`(ロード・検証)・`ui/`・`stats/` の分割と依存方向 | **coreはDOM・React非依存を厳守**(W1・W3の土台)。W3の確率計算もcore/statsに置く |
| A3 | 状態管理と履歴: GameStateイミュータブル+行動=遷移関数。アンドゥの実現方式(状態履歴保持 vs シード+行動列の再実行) | W3のため**GameStateは任意の途中状態から構築・シリアライズ可能**にする |
| A4 | 乱数設計: シード指定可能なPRNG注入と、乱数消費順の固定化(リプレイ・統計の再現性) | みだれぬい等が乱数をいくつ・どの順で引くかを実装仕様として明文化する |
| A5 | データ読み込み方式: ビルド時バンドル vs 実行時fetch(F7エディタとの兼ね合い) | W1ではアプリへの同梱になるため、読み込み層を抽象化しておく |
| A6 | 行動ログ・リプレイ形式: 「シード+行動列」を正とするか、各ターン結果込みとするか | |
| A7 | 配布形態: ローカル起動 / GitHub Pages | W1の布石としてPWA化の要否もここで判断 |
| A8 | **モバイル移植戦略(W1)**: PWA / Capacitor(WebViewラップ) / React Native のいずれを見込むか | 初期実装をどれにしても動くよう、coreの分離(A2)とレスポンシブUI(4.4)を担保する。ストア配布まで見込むならCapacitorが有力 |
| A9 | **実装前提の設定化(W2)**: 職人レベル・コツ・パッシブ・針を `SimulatorConfig` としてcore APIに定義(デフォルト=SPEC §1.1) | 初期UIは針のみ露出。他はconfigのデフォルト値として実装 |
| A10 | **計算機モード向けAPI(W3)**: 途中状態入力 → 確率・期待値算出のインターフェース設計(実装はしない。APIの形だけ将来を見込む) | F9ソルバーと共通基盤にする |

### 4.3 データ設計

- game-params.json / damage-tables.json / needles.json / skills.json / concentration.json の具体的なフィールド定義(スキーマ)
- recipes.csv → 内部モデルへの変換規則とバリデーション(部位とrows/colsの整合、マス数と評価境界の対応、power_orderトークン検証)
- スキーマはSPECの数値を機械可読へ転記する工程を兼ねる。転記ミス防止のため、テスト(4.5)で仕様書の数表と突き合わせる

### 4.4 UI詳細設計(差分のみ)

基本レイアウトは合意済みモックを正とし、以下の未確定部分のみ詰める:

1. 特技の対象選択フロー(単マス/複数マス起点選択/対象不要のみだれぬい・精神統一等の操作差)
2. 結果画面(できのよさ表示・誤差内訳)
3. 統計モード画面、パラメータエディタ画面
4. 練習/検証モード切替とパワー予告表示の切替UI
5. **タッチ操作・スマホ縦画面での成立性(W1)**: クリック前提の操作をタップで置き換えられるか

### 4.5 テスト計画・準備

- フレームワーク: Vitest(A1と同時確定)
- コアのユニットテスト方針:
  - ダメージテーブル全セルとダメージ式の一致(仕様書§3.2の全数値をテストケース化)
  - 丸め規則(正負・括弧ごと)、糸ほぐしの負ダメージ・回復上限
  - 会心(2倍・基準値頭打ち・会心率式の各補正項)
  - 布特性(発動ターン5,9,13…、再生布の対象選定・タイブレーク、虹布の交互と端数切り上げ、光布の候補条件)
  - 評価境界(マス数4種×境界値の境界テスト)、黄色ゲージ外ペナルティ
  - 集中力(開始値、不足時、自動回復の条件と1回制限)
- シード固定による決定論テスト(同一シード+同一行動列→同一結果)
- 統計的テスト(大量試行で基礎値の等確率性・会心率の近似を検証)
- データ検証テスト(JSONスキーマ・recipes.csvバリデーション)
- 実機突き合わせシナリオ(ユーザーが実機データと比較する受け入れ観点)

## 6. 委譲運用ルール(コンテキスト消費対策。2026-07-06 承認)

サブエージェントの使用量(テストの反復実行・設計文書のコールドリード・中断からの再開)が全体消費の大半を占めるため、以下を委譲の標準とする:

1. **テスト実行の絞り込み**: 開発中は変更に関係するテストファイルのみ実行(`npx vitest run <対象> --reporter=dot`)。全スイート(unit+stats)・build・eslint は完了時に1回だけ実行する
2. **文書全読みの廃止**: エージェントに docs/ 一式を読ませず、Fable5 がタスクに関係する仕様の節をプロンプトへ直接引用する
3. **タスク粒度の縮小**: 1エージェント=1修正テーマ(ツール呼び出し30回規模)。セッションリミット付近では大きなエージェントを起動しない
4. **定型作業のHaiku化**: データ転記・単純置換・機械的なテスト追加は model: haiku で委譲する
5. (予定)engine.ts のモジュール分割(特技・布特性・ターン進行)を次の大型改修の直前に実施し、部分読みで済む構造にする
