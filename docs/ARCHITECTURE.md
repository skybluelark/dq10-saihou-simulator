# ドラクエ10 さいほうシミュレータ 方式設計書 (v0.1)

作成日: 2026-07-03 / 状態: A1〜A10 承認済み (2026-07-03)
関連文書: [SPEC.md](SPEC.md)(ゲーム仕様) / [DEVELOPMENT_PLAN.md](DEVELOPMENT_PLAN.md)(開発計画)

## A1. 技術スタック

- TypeScript + React 18 + Vite
- テスト: Vitest
- 状態管理ライブラリは使用しない(`useReducer` + core関数)
- CSSは軽量構成(CSS Modules 想定)

## A2. モジュール構成

```
src/
  core/   ゲームエンジン(状態遷移・判定。純関数のみ)
  data/   スキーマ型・ローダ・バリデーション
  stats/  モンテカルロ実行・集計(将来対応: F8統計モード・F9ソルバー・W3計算機の共通基盤。初期は未実装)
  ui/     Reactコンポーネント
```

- 依存方向: `ui → stats → core`、`ui → data → core`。**core は React・DOM・fetch に一切依存しない**(W1スマホ移植・W3計算機モードの土台)。
- 依存制約は ESLint ルールで機械的に強制する。
- stats/ は初期スコープでは実装しないが、モジュール境界として予約する。

## A3. 状態管理と履歴

- `GameState` は JSONシリアライズ可能なイミュータブルのプレーンオブジェクト。
- 行動は純関数 `applyAction(state, action, rng) → { state, events }`。events はUI表示・ログ用のイベント列。
- アンドゥ: 状態履歴スタックを保持(状態は小さいため全ターン分保持可)。
- リプレイ・統計: 「シード+行動列」の再実行で実現(履歴スタックと併用)。
- W3(計算機モード)用に、途中状態から直接構築する `createStateFromSnapshot()` をコアAPIに含める。

## A4. 乱数設計

- 注入式の `Rng` インターフェース+シード指定可能な軽量PRNG(mulberry32)。
- **乱数の消費順(確定版・M1実装済み。T9ゴールデンテストで固定)**(シミュレータ内の再現性のための取り決めであり、実機の乱数再現ではない):
  1. createSession時: 針の開幕効果判定 1ロール(開幕効果のない針は消費なし)
  2. ターン開始時(そのターン最初の行動適用時。しあげるは消費なし):
     [「？」の抽選 nextInt(5)(？のターンのみ。精神統一固定中は消費なし)] →
     [光布の発光対象選定 nextInt(候補数)(光布の発動ターンで候補ありのみ)] →
     [集中力自動回復判定 next()(未判定かつ残≤10のみ)]
  3. 行動本体:
     - 縫い(単/複数マス): 対象マス順に [基礎値 nextInt(7)] → [会心判定 next()]。布外・欠けマスは消費なし。critx2ターンも会心判定ロールを消費する
     - みだれぬい: [対象選択 nextInt ×4(部分Fisher-Yates)] → 打ち順(2/1/1/0.5倍)に [基礎値]→[会心] ×4 → 会心適用後の値で降順ソートして適用
     - 糸ほぐし: [基礎値 nextInt(4)] のみ(会心判定なし)
     - ぬいパワーシフト: [変更先 nextInt(4)]。精神統一・しつけがけ・無我の境地は消費なし
  4. 布特性(行動適用後・同ターン内): 再生布の発動ターンのみ [対象タイブレーク nextInt(同率数)(同率1マスなら消費なし)] → [回復量 nextInt(5)]。虹布は消費なし
  5. ターン終了時: [必殺チャージ判定 next()](与ダメージ>0かつ未チャージかつ必殺未使用のみ)

## A5. データ読み込み

- `DataProvider` インターフェースで抽象化(W1のアプリ同梱にも対応)。
- JSON類(game-params / needles / skills / concentration)はビルド時バンドル(ダメージ値はコアが計算式で算出するためデータファイルなし — DATA_DESIGN §2)。
- recipes.csv は実行時 fetch+自作の軽量CSVパーサ(フォーマット固定のため外部依存なし)。**CSV編集→ブラウザ再読み込みだけで反映**でき、再ビルド不要。
- F7 パラメータエディタの編集値は「バンドル値+localStorage のオーバーライド」として重ねる。
- W1でのアプリ内レシピ入力は、DataProvider 配下に端末内ストレージ(IndexedDB)実装を追加して対応。CSVはマスタ+インポート/エクスポート形式として維持(DATA_DESIGN §8)。

## A6. 行動ログ・リプレイ形式

- 正とする形式: 「シード+SimulatorConfig+レシピid+行動列」のコンパクトJSON(テキスト共有可能)。
- 画面表示用の人間可読ログ(例: `T4: 2倍ぬい → 中央 -28`)は上記から導出する。
- バージョン間の仕様差検出用に、最終結果値のチェックサムをオプションで含められるようにする。

## A7. 配布形態

- GitHub Pages(静的配信)。開発時は `npm run dev`。
- file:// 直接起動は fetch 制約のため非対応。
- PWA化(オフラインキャッシュ・ホーム画面追加)は vite-plugin-pwa で後付けできる構成とする(W1の第一歩)。

## A8. モバイル移植戦略(W1)

- **PWA → Capacitor の段階移行**を想定。Capacitor は Web資産をそのまま Android/iOS アプリとして包めるため、ストア配布まで到達可能。
- React Native は UI全面書き直しになるため採用しない。
- 初期実装で担保するのは「core の分離(A2)」と「レスポンシブUI」のみ。移植技術の最終確定は W1 着手時。

## A9. 実装前提の設定化(W2)

コアAPIは常に `SimulatorConfig` を受け取る。デフォルト = SPEC §1.1。

```ts
interface SimulatorConfig {
  level: number;                 // デフォルト 80
  kotsu: boolean;                // デフォルト true
  passives: { critUp: boolean; hissatsuUp: boolean };  // デフォルト true
  needle: { type: NeedleType; stars: 0 | 1 | 2 | 3 };
}
```

- 初期UIで露出するのは needle のみ。他はデフォルト値のまま(W2でUI追加)。

## A10. 計算機モードAPI(W3)

- 型定義のみ先行して用意し、実装は M5/M6:

```ts
evaluate(state: GameState, config: SimulatorConfig, options: EvaluateOptions)
  → { starProbabilities: Record<Star, number>; recommendedAction?: Action }
```

- stats のモンテカルロ基盤を共用。途中状態の入力は A3 の `createStateFromSnapshot()` を使う。

## 非機能要件の決定 (2026-07-03)

| # | 項目 | 決定 |
|---|---|---|
| N1 | 動作環境 | PC最新ブラウザ(Chrome/Edge/Firefox)を対象。**スマホ版UIは表示領域を考慮しW1で抜本的に別設計とするため、初期UIではスマホ表示を考慮しない**(2026-07-06 更新)。coreのUI非依存(A2)は維持し移植性を担保 |
| N2 | オフライン動作 | 外部通信に依存する機能は作らない(全データ同梱)。完全オフライン化(PWAキャッシュ)はW1で追加 |
| N3 | 統計モード性能 | **統計モード(F8)は初期スコープから除外**(固定行動列の再生ではシミュレートの意味が薄いため)。盤面に応じて手を選択する戦略ポリシーの実装と合わせて将来対応。性能要件はその時点で再定義 |
| N4 | ローカル保存 | **UI上で選択可能な項目は localStorage に自動保存し、次回起動時に同じ状態に復元する**。現時点の対象は針(種類・★数)のみ。UI詳細検討で項目追加の可能性あり |
