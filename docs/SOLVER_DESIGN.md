## S2. 問題のモデル化

- 確率的な逐次意思決定問題(MDP)。状態 = `GameState`(コア既存)、行動 = `(skillId, anchor)` + `finish`、遷移 = `Engine.applyAction`(探索用RNG注入)。
- 報酬はしあげる時点の judge 結果のみ: `star3 → 1 / それ以外 → 0`。
- 厳密解(全深さ expectimax)は分岐数(候補 20〜60/手 × 縫いマスごと基礎値7通り×会心2通り × 深さ約20手)から計算不可能。**1手の厳密な期待値計算+先読みはモンテカルロ近似**で構成する。
- 推奨の起点は `beginTurn` 済みの状態(当ターンのパワー・光布発光・自動回復が確定した後)。将来ターンの「？」抽選・布特性・必殺チャージ等は確率イベントとして扱う。

## S3. 計算パイプライン

### S3.1 Stage A: 候補列挙と解析的1手評価(決定的・数十ms)

**候補列挙**(`stats/actions.ts`):

- 全アクティブ特技 × 全マスをアンカーとして展開し、`finish` を常に候補に含める。
- ライン系はアンカー自動置換(`clampAnchorForPattern`)適用後の**対象マス集合が同一になる候補を重複排除**。
- 除外規則: 実効コスト(`effectiveCost`)> 残り集中力 / 対象不成立(コアの `hasValidTarget` 相当)/ 縫い系で対象全マスが残り≤0 / 糸ほぐしで対象マスの累積=0 / しつけがけで対象マスが既にしつけ中または残り≤0 / 無我の境地はチャージ保持中かつ未使用時のみ。
- 精神統一・ぬいパワーシフトは常に候補(価値判断は評価関数に委ねる)。

**1手の結果分布**(`stats/distribution.ts`): 乱数を使わず厳密に計算する。

- 縫い1マス = 基礎値7通り(12〜18等確率)×会心有無の離散分布。ダメージ・丸め・頭打ちはコアと同一規則(`sewDamage` + 会心2倍 + **会心のみ残りで頭打ち**・非会心は縫いすぎ許容・残り≤0のマスは会心判定なし)。会心率は `Engine.critRate`(ねらい・シフト会心・発光・虹・無我込み)。
- 糸ほぐし = 出目6〜9等確率、`hogushiDamage`、初期状態頭打ち。
- 複数マス特技はマスごとに独立分布の合成。みだれぬいは「各マスが各倍率 {2,1,1,0.5} をそれぞれ確率 1/n で受ける」周辺分布で近似(n=マス数。厳密な結合分布はロールアウト側が実エンジンで担保)。
- 会心の頭打ち(残りちょうどで止まる)が分布に自然に反映されるため、「ねらいぬい+高会心率=残りピッタリ狙い」が解析的に評価される。

**静的評価関数 V(state)**(`stats/evaluate.ts` + `stats/finishing.ts`): ★3到達見込みのスカラー近似。候補の1次ランキングとロールアウトポリシーに使う。

1. **マス別仕上げテーブル**: 残り値 r ∈ [−30, +30] ごとに(期待最終誤差評価値, 所要手数, 所要集中力)を推定。「仕上げ操作集合」(単マス縫い各種・ねらいぬい会心頭打ち・糸ほぐし・放置)×使用可能パワーの**2手までのDP**(厳密PMF使用、`cellErrorScore` 規則で採点)。しつけ補正×無我の4変種をソルバー初期化時に事前計算。
2. **削り工程(r > fineLimit)の効率モデル**(M③で改訂): 所要集中力 = 残り作業量 ÷ 達成可能効率。効率 = `workEfficiencyBase` × 今後Hターンのパワー係数平均(固定・強制・サイクル。「？」は期待係数1.2)。これにより**ダメージ/集中力の効率**が候補スコアの勾配として働き(みだれ・ライン系・強パワー集中が優位)、精神統一・シフトの価値も係数平均を通じて評価される。期待誤差は仕上げ域(r=8..16)の平均値(定数)とし、勾配は効率項が担う。
3. **集計**: `margin = ★3境界(boundsFor) − Σ期待最終誤差`、`不足 = max(0, Σ所要集中力 − 残り集中力)` として `V = σ(margin/s0) − λ×集中力不足`(σ=シグモイド)。しあげる候補は judge 直接評価: **★3なら V=1 で最優先、★3未達なら V=−1**(★3未達のしあげる価値は厳密に0であり、望みのある継続候補が常に優位になるよう下限に置く。0にすると序盤にVが負の縫い候補より「即しあげる」が優先される欠陥がある — M③で実測し修正)。係数(s0, λ, workEfficiencyBase 等)は `solverParams` に集約しM④で校正。

候補のスコア = 1手後分布上の E[V]。対象マス以外は不変なのでマス周辺分布の合成で高速に計算できる。シフトは結果4通りの期待値。

### S3.2 Stage B: モンテカルロ・ロールアウトで★3率推定

- Stage A 上位 K 候補(既定8)について、候補実行後「グリーディポリシー(Stage A の1手評価で選択)で終局まで」を試行。遷移は `Engine.applyAction`/`beginTurn` をそのまま使用(探索用シード決定的払い出し: base seed+候補番号+試行番号)。無限ループ保険として1試行の行動数上限(既定100)。
- 集計: 候補ごとに (試行数 n, ★3回数 w, 誤差合計, 残集中力合計)。★3率は Wilson 95%信頼区間つきで報告。
- **racing方式**: 首位と信頼区間が重なる候補へ追加試行を集中配分し、明確に劣後した候補は打ち切る。
- 最終推奨 = ★3率最大の候補。`finish` も同じ土俵で比較される(=しあげ時機判断を内包)。

### S3.3 Stage C: 公称プラン(しあげるまでの手順概算)

- 1手目=推奨手に固定し、以降はグリーディポリシーを**公称乱数(NominalRng)**で終局まで決定的に実行した行動列を「ターン / パワー / 特技+対象 / 予想盤面」の表として出力。末尾に予想誤差評価値と★数。
- **NominalRng**: `nextInt(n) = floor((n−1)/2)`(中央値。基礎値=15、糸ほぐし=7相当、再生回復=14)、`next() = 0.9`(会心は率>90%のほぼ確定時のみ発生。自動回復・必殺チャージは発生しない)。「？」は中央値則で「つよい」になる。実エンジンをそのまま使うため遷移ロジックの二重実装を避けられる。
- 公称プランでは結果が乱数依存の特技(みだれぬい・ぬいパワーシフト・無我)は候補から除外する(1手目が該当する場合は2手目以降のみ除外)。
- UI には「順調に進んだ場合の目安であり、出目により変わる」旨を注記。実際にずれたら次の推奨実行でプランが引き直される。

## S4. anytime実行(再実行による精度向上)

- ロールアウトは独立試行のため、**状態キーごとに (n, w) を保持し再実行分を合算**するだけで信頼区間が狭まる(√n オーダー)。
- 状態キー = `GameState` の正規化シリアライズ + config + ソルバー版数のハッシュ。ユーザーが行動すればキーが変わり集計は自然に破棄。
- 探索シードは実行回数を含めて決定的に払い出す(同条件の再現性・テスト可能性)。
- 時間予算: 初回既定1000ms(Stage A 数十ms+残りをStage B)。再実行1回につき+2000ms(仮。M③のUI検討で確定)。

## S5. モジュール構成・API

```
src/stats/
  types.ts         ソルバー型・solverParams(調整パラメータ集約)
  actions.ts       候補列挙・重複排除・除外規則
  distribution.ts  1手の結果分布(マス別PMF)
  finishing.ts     マス別仕上げテーブル(事前計算DP)
  evaluate.ts      静的評価 V(state)
  greedy.ts        1手グリーディ選択(ロールアウトポリシー兼用)
  rollout.ts       ロールアウト実行・集計
  anytime.ts       サンプル合算・racing配分・信頼区間
  nominal.ts       NominalRng・公称プラン生成
  solve.ts         パイプライン統括(A10 evaluate API の実装)
  worker.ts        Web Worker エントリ(M③)
```

- 依存方向は A2 どおり `ui → stats → core`。stats は React/DOM 非依存の純TS。データ(skills等)は `EngineData`/`Engine` 経由で受け取る(stats → data 依存は作らない)。
- A10 の計算機モードAPI `evaluate(state, config, options) → { starProbabilities, recommendedAction }` は `solve.ts` を実体とし、W3 と共用する。
- **コアへの追加(挙動・乱数消費順の変更なし)**: `Engine.effectiveCost` / `resolveTargets` / `cellAt` を public 化、会心率計算を `critRate()`(乱数を消費しない率計算)として公開し `rollCrit` が内部利用、特技一覧の getter を追加。
- Worker 実行(M③): `GameState` はシリアライズ可能なのでそのまま postMessage。進捗(暫定順位・信頼区間)を逐次通知。

## S6. 性能目標

- 初回応答: 1秒以内(Stage A ≤100ms + ロールアウト数百試行)。
- ロールアウト1試行 ≈ 残り約20手 × 候補評価。1秒で「上位8候補 × 合計500試行」程度を目標とし、未達なら候補評価の簡略化(粗い事前フィルタ)で調整。
- 参考: `applyAction` は9マス盤面でµsオーダー。ボトルネックは想定上グリーディの候補全評価。
- **M③時点の実測: 約30試行/秒**(1試行=フルゲーム約29手。ポリシー修正前の計測値は即しあげる欠陥による過大値だった)。M④のホットパス最適化(非対象マス評価の再利用、仕上げ域平均の事前計算、候補の粗フィルタ等)で目標へ引き上げる。

## S7. テスト(TEST_PLAN T13)

1. 候補列挙: 重複排除(ライン系クランプ)・除外規則・無我/しあげるの条件を網羅。
2. 分布計算: RNGスタブで基礎値・会心の全分岐を列挙し、`Engine.applyAction` の実結果とPMFが**厳密一致**すること(統計テスト不要の決定的検証)。
3. 評価・グリーディ: 明白な局面での妥当性(全マス誤差0なら finish を選ぶ / ★3確定盤面で縫い続けない / 集中力不足の候補を選ばない)。
4. 決定論: 同一状態+同一シード+同一予算 → 同一の推奨・同一の集計。
5. anytime: 合算の正しさ(2回×N/2 = 1回×N と同一集計)。
6. 統計(statsスイート): 推奨ポリシーの★3率がベースライン(ランダム合法手・単純ルール)を有意に上回ること。公称プランが終局までに到達すること(全レシピ)。

## S9. v1の既知の近似・制限(将来改善候補)

- 静的評価はマス独立近似+需給ヒューリスティック(結合分布・深い先読みはロールアウトが補正)。
- みだれぬいの1手評価は周辺分布近似。
- 布特性(再生・虹・光)の将来価値は V に明示項を持たず、ロールアウト側でのみ厳密に反映される。
- 公称プランは中央値決め打ちの目安(乱数依存特技を除外)。
- 探索の深読み(depth-2 expectimax)や終盤の厳密DPは、④の精度評価で不足が判明した場合の拡張枠。
