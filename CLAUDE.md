# dq10-saihou-simulator 作業指示書

DQ10さいほう職人ミニゲームのシミュレータ(TypeScript + React + Vite + Vitest)。
本書はメインセッションのモデル(Opus等)が品質を維持するための必須指示。**変更作業の前に必ず本書に従うこと。**

## 1. 文書体系(正の所在)

| 文書 | 役割 |
|---|---|
| docs/SPEC.md | **ゲーム仕様の正**。数値・確率・挙動はすべてここが基準 |
| docs/ARCHITECTURE.md | 方式設計。特に **A4 乱数消費順は変更厳禁**(リプレイ互換の根幹) |
| docs/DATA_DESIGN.md | データスキーマ・recipes.csv検証規則 |
| docs/TEST_PLAN.md | テスト体系(T1〜T12)。仕様書の数表=テストの期待値 |
| docs/DEVELOPMENT_PLAN.md | マイルストーン・委譲運用ルール(§6) |
| data/recipes.csv | ユーザー管理のレシピデータ(勝手に書き換えない) |

## 2. 絶対的なルール

1. **仕様(SPEC.md)の数値・確率・挙動をユーザー確認なしに変更しない**。ネット上の情報は不正確前提であり、仕様の確定はユーザーの実機実測のみが根拠になる。
2. **仕様の不明点・曖昧さは推測で実装しない**。番号付きの質問リストにしてユーザーへ確認する。どうしても仮定を置く場合は SPEC §6「仮定一覧」へ記録し、game-params.json で調整可能な形で実装する。
3. 仕様を変更・追記したら SPEC.md の**版数を上げ、更新履歴に1行追加**する(このプロジェクトの慣行。他文書も同様)。
4. **乱数消費順(ARCHITECTURE A4)を変えない**。挙動修正でやむを得ず変わる場合は、T9ゴールデンテストの差分が「その修正で説明できる」ことを確認してから `-u` で更新する。説明できない差分が出たら止めてユーザーへ報告。
5. `src/core` は React/DOM/fetch に依存しない純TS(ESLintの依存方向ルールで強制)。実装前提(Lv80・コツ等)は `SimulatorConfig` 経由で、ハードコードしない。
6. git commit はレビュー完了後にメインセッションが行う(サブエージェントには commit させない)。メッセージは英語+`Co-Authored-By: Claude <モデル名> <noreply@anthropic.com>`。

## 3. 委譲運用(DEVELOPMENT_PLAN §6 準拠)

メイン=設計判断・タスク分解・成果物レビュー。実装は sonnet、定型作業(データ転記・単純置換・機械的テスト追加)は haiku へ委譲する。設計と実装が密接な箇所・高難度箇所はメインが直接対応してよい。

委譲プロンプトの必須要素:
- **関係する仕様の節をプロンプトに直接引用する**(docs一式を読ませない — コンテキスト消費対策)
- 変更対象ファイルと変更禁止領域(docs/ と data/ は原則サブエージェント変更禁止)
- 開発中のテスト範囲: **変更に関係するテストファイルのみ** `npx vitest run <対象> --reporter=dot`。全ゲートは完了時1回
- git commit しないこと、報告事項(変更一覧・テスト結果・判断に迷った点)
- 環境注意(下記§6のPATH前置き)

タスク粒度は 1エージェント=1修正テーマ(ツール呼び出し30回規模)。**セッションリミット付近で大きなエージェントを起動しない**(中断→再開はトランスクリプト再読込で高コスト)。

## 4. 品質ゲート(コミット前に必ず全部)

```
npm test            # unit 全パス(370件前後)
npm run test:stats  # 統計テスト 13件(シード固定・3σ)
npm run build       # tsc + vite build
npx eslint src tests
```

- UI変更時は加えて**ブラウザ実確認**(launch.json の "dev"、ポート5173)。操作フロー(特技選択→マスタップ→再タップ実行)が通ること、コンソールエラーがないこと。
- `npm run test:quick` = dotレポータの圧縮出力版(開発中の確認用)。

## 5. レビュー観点チェックリスト(サブエージェント成果物に対して)

1. SPECの数表・規則との一致: ダメージ式と丸め(正方向 `Math.ceil`、糸ほぐしは負値・会心なし・初期状態頭打ち)、会心(2倍→基準値頭打ち)、評価境界(マス数4種)、布特性(T5初回・4ターンごと)、アンカー自動置換(SPEC §3.3)
2. 差分が指示した範囲に収まっているか(無関係な変更・雛形設定の変更がないか)
3. リプレイ互換: シード+行動列の再実行同一性(T9)が維持されているか
4. スナップショット/ゴールデン更新の妥当性(§2-4)
5. テストが「実装に合わせて」ではなく「仕様に合わせて」書かれているか(期待値の根拠がSPECにあるか)
6. サブエージェントの報告にある「判断に迷った点」「解釈した点」を必ず精査し、仕様に関わるものはユーザーへの質問に昇格する

## 6. 環境注意(このマシン固有)

- **PATH**: シェルが Node.js インストール前のPATHをキャッシュしている。npm/node を使う各コマンドの先頭で毎回:
  `$env:Path = [Environment]::GetEnvironmentVariable('Path','Machine') + ';' + [Environment]::GetEnvironmentVariable('Path','User')`
  (委譲プロンプトにも必ず記載する)
- プレビューは `.claude/launch.json` の "dev"(npm.cmd フルパス指定済み)。プレビュータブは viewport 0×0 になることがあり **preview_click が効かない/スクリーンショット不可**の場合がある。UI操作確認は `preview_eval` で `element.click()` を使い、**クリックと状態読み取りを別々の呼び出し**にする(同一eval内では状態反映前の値を読むため)
- ファイルはUTF-8(recipes.csv はBOM付き)。data/sample/*.csv は Shift-JIS(読むときは CP932 デコード)
- git の改行警告(LF→CRLF)は無害。コミット成功は `git log` で確認

## 7. ユーザーとの進め方(合意済みスタイル)

- 段階承認スタイル: 仕様に関わる変更は「SPEC更新→ユーザー確認(必要時)→実装」。UIの使い勝手はユーザーが実機と突き合わせて都度指摘してくるので、指摘→SPEC§4.3反映→実装の順
- 複数の修正依頼が続く場合、テスト等の重い検証は**まとめて最後に1回**でよい(ユーザー了承済み)
- 報告は結論から。委譲先の報告を鵜呑みにせず、テスト再実行と差分確認をメイン側で行ってから完了報告する
